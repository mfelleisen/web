6 — The GUI

Due Thursday, 03 November 2022, 11:59:59pm

Delivery Place the product of this week’s milestone into your git repo as follows:

  • for the Programming Task, place interactive-player.PP into Maze/Players/;

    Some languages demand a specific organization within the file system. In that case, you may place a link to interactive-player.PP into Maze/Players/

  • for the Testing task, place xgame and Tests/ into a top-level repo directory named 6

All previous comments on code files apply.

Programming Task Design and implement a graphical user interface (GUI) for a player. The mechanism should primarily serve as a tool for soliciting a strategic choice when the referee grants the player a turn. You may optionally add a text field that informs players of events, such as setup and wrap-up; alternatively, print those alerts to a console.

The screen shot does not display the entire GUI interface. The GUI should support two different forms of gestures: clicks on buttons and mouse clicks. Specifically, the GUI should allow players to

  • some gesture of your choice to rotate the tile before insertion;

  • click on buttons for indicating row and column shifts; and

  • mouse clicks for selecting a new place for the avatar.

See figure 1 for what such an arrangement may look like; yours may look quite different as long as it is intuitive to our TAs.

Make sure to make the interface reasonably robust. For example, don’t allow users to select a place for the avatar before a row or column has been shifted and the spare tile has been inserted.

Do not implement an undo functionality. That is, implement the policy that once users choose an action on an item, they commit to it for good. (No, this is not how we would deliver such a GUI, but the constraint simplifies your life.)

Testing Task Create a test harness named xgame. The harness consumes its JSON input from STDIN and produces its results to STDOUT. Create three tests and place them in the specified Tests/ folder.

Keep in mind our Maze.Com, a Plan while you work on a design task.

Its inputs consists of a game RefereeState and a PlayerSpec. The test harness creates a series of colored players that are going to use the specified strategies. It then hands these players and the given state to the referee, which runs the game to the end and computes the winner(s). (The details will depend on your referee interface and data representations.)

The output is an array of alphabetically sorted (string<?) Colors, denoting the winners.

    A RefereeState is like a State, except that its

      "plmt" field contains an array of RefereePlayers.

    

    RefereePlayer is an object with four fields:

      { "current" : Coordinate,

        "home"    : Coordinate,

        "goto"    : Coordinate,

        "color"   : String }

    

    INTERPRETATION A player object describes a player's current location, its home,

      its current target coordinate, and the Color of its avatar.

    

    CONSTRAINT The colors of two different RefereePlayers are distinct.

    

    A PlayerSpec is a JSON array of 2-element JSON arrays:

      [[Color, Strategy], ..., [Color, Strategy]]

    

    INTERPRETATION Each 2-element array specifies which strategy the

      player of the given color employs.

    

    A Strategy is one of: Riemann, Euclid.

    

    GLOBAL CONSTRAINT The array of RefereePlayers (in RefereeState)

      contains as many players as the PlayerSpec. Extracting the color

      attributes from all RefereePlayers and all PlayerSpecs

      yields identical sets.
