What is a software system? What is it like to create one from scratch?

Software developers rarely get to build a software system from scratch. When they do get this chance, they could try to build the system “the right way.” Experiences teaches us, however, that external stresses like demo deadlines or fund-raising show-and-tells upset the best plans. Or, as Moltke said “no battle plan survives contact with the enemy.”

The Programming Language

In most companies, software developers work under numerous constraints. To start with, managers will insist on the use of certain programming languages. Some may even restrict developer to use certain IDEs. They may also wish to impose certain processes (“agile” is a beloved buzzword), ask for certain architectural styles (say, micro services), or impose other constraints.

The instructors of software development course have the freedom to combine various elements. Students need to understand the fundamental principles of a software system, how to tease out the essential aspects from a “problem statement,” and how to translate these insights into a plan.

Once these ideas are understood, it takes discipline to stick to a socially responsible approach with technical skills.

7 Software Systems

A software system consists of many interacting pieces of software. Over time the pieces have been called by different names: component, module, package, plug-in, (micro)service, and more. These names reflect our growing understanding of the nature of software systems. While programmers early on thought that software is designed, implemented, and then deployed for good—like any other engineering product, say, a bridge—we now understand the extremely dynamic nature of software. Software systems evolve over time, which is one of the key premises of the preceding parts of this book. And, deployed software systems change while they execute: they configure and re-configure themselves; they load and unload pieces dynamically; and they connect and disconnect external pieces.

Consider the run-time cycle of a reasonably complex software system, say, a web server, an IDE, or a distributed game app. When launched, it configures itself according to instructions from the command-line and/or a configuration file. During a configuration process, the system loads components, passes on configuration information, and in many cases, goes through a period of instability. Components may become available in a concurrent manner, so any individual component cannot assume that whatever services it needs are available. Once all configuration activities have completed, the system is in a steady state, delivering services in response to requests.

In response to external requests, software systems may load or unload components. Some of these run-time extensions dynamically integrate modules or classes into the running code. Others execute components as separate processes—to sandbox any effects—and use the results via operating systems facilities (files, pipes). Yet others connect to external components. Think of an IDE; in many cases, a developer may ask such a system to add a tool. An operating system will unload apps that have finished their task. A stock ticker may connect to an endpoint at some market information provider’s site.

No matter how the software system is extended, new elements introduce new complications and the potential for failure. Adding a class or module may interfere with the internals of the running system just by shadowing an existing name or by breaking a logical invariant. Programming language designers have learned to help with the former, but the latter shows up time and again when operators hot-swap one version of a module for another because it is impossible to take down the system. While spawning a separate operating system process protects a software system from direct effects, say, address interference, it also means that the results have to be communicated and processed at a low-level, say, a byte-stream—and that poses its own problems. The same kind of problem shows up when a system connects to distributed components over some network protocol.

When a critical component fails, a software system may have to shut down; when a peripheral component fails, it may suffice to unload or disconnect from this component. We know that all software comes with flaws. We also know that people have come to depend on the continuous execution of some software systems, for example, the phone network; such systems should never go down and, if they do, they must shut down without doing additional harm. And finally, we know that software systems consist of many components. So putting this together means software developers must prepare their creations to recover from partial failures and to exit gracefully on fatal failures.

On some occasions, operators must take down software systems. People typically speak of “maintenance” in such cases and mean many different things. Whatever the reason, when a software system receives the request to shut down, it must do so properly. It should save any data it may need for a re-start; it should terminate any external processes that it launched; and it may have to release allocated resources.Modern operating systems properly manage resources, but some software systems must run on impoverished platforms.

Let’s summarize this analysis because a responsible software developer must plan for every one of these aspects of software systems:

  • Any interesting software system comes with a start-up configuration.

  • During the start-up phase, a software system is unstable.

  • Once the execution of a system enters its stead-state phase, external requests may call for dynamic extensions and re-configurations.

  • Dynamically changing systems may fail, partially or totally, due to bugs.

    • The failure of peripheral components should not tear down the entire system.

    • If a critical component breaks, a software system may have to shut down—gracefully.

  • While some systems are expected to run forever, others must be prepare to stop on occasion.

As you continue to practice your software development skills, keep these essential aspects in mind. Every one of them calls for acting in a socially responsible manner, both with respect to users but also with respect to present and future colleague.
