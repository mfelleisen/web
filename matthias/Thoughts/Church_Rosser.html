Church Rosser🔗

Matthias Felleisen

Jun 2 2024

Changed in version 1.0: Sun Jun 2 14:04:32 EDT 2024, initial release

Over the past couple of months, several people have brought up the Church-Rosser property in conversation, and I was somewhat amazed at the uninformed opinions that these people reported.

As someone who has worked on his fair share of Church-Rosser theorems, I figured I would write down what I learned in this regard over the course of 40 years.

Church and Rosser Prove that their Calculus is Sound🔗

Neither Church nor Rosser designed a programming language. The λ-calculus per se is a programming language only as much as Turing machines are. So when someone says “people in 1936 knew how to deal with property X in programming languages via the Church-Rosser theorem” then this statement is (1) anachronistic and (2) wrong.

Church and Rosser understood that writing down n axioms A to generate an equational proof system did not imply that working out proofs in this system would make sense. They had to show that

A ⊢ true = false

was impossible. Because once we can prove this equation, we can prove every other equation, but a proof system is about discriminating against falsehoods.

A proof system that validates only true statements is sound.

Church and his colleagues had come up with several such systems, which Barendregt (North Holland, 1980) labels as λI, λK, λβ, λβη, and so on.

From λ-Calculus to Programming Languages🔗

In the late 1950s and 1960s our colleagues started to realize that the λ-calculus is an interesting basis for discussing aspects of programming languages. One aspect was scope; when is a variable x bound by a λ x. in its context. Another one is language syntax; why not include functions as values in a programming languag and mark them with the keyword λ or lambda. Once Scott had figured out how to describe The Truth behind the syntactic λ-calculus, Strachey could use the λ notation as a framework for writing down a semantics of (pieces of) existing programming laguages—but the meta-notation really just referred to elements of domains.

And finally, Bohem, McCarthy, Landin, Reynolds, Morris, and others decided to design programming languages directly based on the λ-calculus. They added literal constants and libraries of functions, plus various conditionals and other forms. They were clever and used evaluation functions (interpreters) often based on abstract machines; Landin’s SECD machine is the most well-known example.

Notably they did not use Church’s axioms to specify the semantics of their designs.

Call-by-name, Call-by-value and the λ-calculus🔗

This last step—designing languages based on λ but using abstract machines to define the semantics—the obvious question how these languages related to the systems of equations generated by Church’s axioms.

The question had become interesting because some language designs used Algol 60’s call-by-name regime, evaluating argument expressions every time a parameter was referenced. Others used call-by-value, meaning the argument expression was evaluated once and for all to a value before the actual function call took place.

Plotkin (1975, TCS) eventually answered this question. Here is the essence of his finding:

  • it is possible to define the meaning of a language via a set of Church-style axioms, which Barendregt calls notion of reduction;

  • but, these notions of reduction are not the semantics;

  • the semantics is a function from programs to values, meaning it is always necessary to identify the subset of values in a language’s set of expressions;

  • different systems of axioms define different evaluation functions;

  • one system of axioms(λn), similar to Church’s, defines what language researchers label a language with a call-by-name convention;

  • a different set of axioms (λv) defines what language researchers label a language with a call-by-value convention;

  • the functions defined via sets of axioms (notions of reduction) are equal to the functions defined by abstract machines that implement the various call-by regimes.

What is the Role of Church-Rosser in this Context?🔗

A semantics from notions of reduction is a relation. To prove that it is a function—that re-evaluations of the same program consistently produce the same result—requires a mathematical proof.

One lemma in this proof is a variant of Church and Rosser’s lemma. That’s it.

What is Truth?🔗

From Church and Rosser’s work, we know that λ ⊢ e = e’ is a proof that e = e’ is true. But what is Truth in the context of programming languages?

Plotkin goes on to define observational equivalence from the evaluation function, following Morris’s dissertation (MIT 1968). Two expressions e and e’ are observationally equal, e ≈ e’, if after plugging e and e’ into any arbitrary program context—an expression with a hole—the evaluation function for the resulting program produces the same outcome.

The Truth for any programming language is vastly larger than the set of equations of an axiomatic syntactic calculus. Indeed, the Truth for such λ-ish languages cannot be axiomatized. Period.

Don’t We Want Equations/Reductions for Optimizing Compilers?🔗

Sure. We want true equations in the compiler. A λ-style calculus might be one way to validate such equations. But, given that the Truth (e ≈ e’) is so much larger than Proof (λ ⊢ e ≈ e’) and given that we want powerful optimizing passes, the axioms play a minor role in this process.

We want as many true equations e ≈ e’ turned into transformations e ⟶ e’ in the compiler as possible. So what we really want is proof techniques for establish Truth between two expressions.

Church-Rosser and notions of reductions are too simplistic.

Use denotational semantics if you want lots of equations. Use logical relations to validate transformations. And there are several other proof techniques.

CR for axioms of notions of reduction is way weak.

Don’t We then Get Nondeterministic Compilers?🔗

Yes.

Depending on when a transformation kicks in, we get different object code. But because they are validated elements of The Truth, running any of these outcomes produces the same value.

And that’s true in reality. I conjecture that all large compilers are non-deterministic translators.

Is it a Good Idea to Define a Semantics via Church-style Axioms?🔗

No.

Ploktin’s work shows that we want an evaluation function first. Write down an abstract machine, perhaps just steps on syntax.

If you need a simple calculus, like λ, develop it after you have worked out the semantic function.

Does this really work? Yes. Starting in 1985, I followed this program, developed λ-style axiom systems for existing semantics, and proved a Church-Rosser theorem for these calculi:

  1. for call/cc-style continuations (LICS ’86, with Kohlbecker, Duba, and Friedman)

  2. a simpler calculus than the one in item 1 (TCS ’92, with Hieb)

  3. for delimited continuations (POPL ’88)

  4. for assignable variables in a λ-calculus-based language (POPL ’87, with Friedman)

  5. a simpler calculus than the one in item 4 (TCS ’92, with Hieb)

  6. for a language with exceptions (Info & Comp ’94, with Wright)

  7. for call-by-need (JFP ’97, with Ariola; ESOP ’12, with Chang)

Items 2, 5, and 7 really show that Church-Rosser is a brittle syntatic property, and we can come up with several different axiom system, for each of which it is more or less elegant to prove CR.

I also had one for futures (POPL ’95, with Flanagan; but it’s not in the paper).

And What About Nondeterminism?🔗

A nondeterministic language may evaluate the same program (and input) to distinct values. The semantics is not a function but a relation.

Designing a CR calculus for such a language should be possible but just a boring exercise in proving some people wrong about the relationship between CR and PL. I have no interest but I think by using sets, it should be possible to do.
