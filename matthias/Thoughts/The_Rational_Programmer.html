The Rational Programmer

Christos Dimoulas & Matthias Felleisen

31 Aug 2021

Mainstream programming language theory rests on many “obvious” claims about programming that, upon closer inspection, turn out to be fabricated.

example 1: finding impedance mismathces between untyped code and type annotations injected into a gradually typed variant of this code

example 2: design

Do you have in mind a design principle for designing  components based on the ratlonal programmer by refinement?

 

If yes I think the refinement recipe you suggest is the following is the following (in the form of questions):

 

1) What is the full state of the system?

 

2) What part of the full state is public?

 

3) What part of the  data is useful to a human user to decide on an action?

 

3.5) What part of the useful data is actually feasible to share between components? (for example time data is meaningless)

 

4) What part of the feasible data can be reconstructed and thus doesn't need to be shared to minimize entanglement?

 

I can see the rational programmer as the omnipotent all-knowing entity that refines its knowledge along these lines. It doesn’t quite fit what we have been doing so far but it is cool to think about this as a strategy for breaking a system into components in a data-driven manner that parallels the design recipe (I am doing this all the time and I have tried to explain to the students in the class but I am not sure I have been very effective).
