2 Grading Process Draft

You must complete the grading within five days---Tuesday night for the Friday morning deadline---of the due date for all project milestones.

All rubrics and self-evaluation forms will be available via the course repo by the first day of class. The instructors and pod leaders have access to the repo.

Ben has allocated a staff-only git repository:

github.ccs.neu.edu/CS4500-F20/code-inspection

When he assigns grading tasks to graders, the assignments (harvested code, self-evaluations) show up in a sub-directory, as specified below. The graders will fill out their code-inspection form and push them back into these repositories.

  1. On the day the assignment is due, Ben will push a self-evaluation sheet into each pair’s repo. The pair has 24 hours to return answers. The goal is to get them to reflect on code and view it from the perspective of a future reader/maintainer.

  2. After the self-evaluation deadline, Ben will push all self-evaluations (complete or not) into the staff-only code inspection repository.

    If we are grading assignment <A> and a pod leader’s CCIS name is <POD> and a grader’s CCIS username is <YOU>, then the to-be-graded code and self-evaluation plus the forms to be filled in will show up in

    github.ccs.neu.edu/CS4500-F20/code-inspection/<A>/<POD>/<YOU>/

    This directory will be populated with a sub-directory per pair, which contains the pair’s complete homework solution ( <A> and their code directory) plus a form file for the grader: form.txt. See below for the shape of the form file and how to use it.

  3. Once homework solutions are assigned to graders, the pod leaders will meet with their pod and jointly grade the repos assigned to the leader. The goal is for the pod to develop a common interpretation of the rubric. The graders will also use Slack to coordinate.

    The pod leaders will communicate with MF to resolve ambiguities.

  4. As a grader inspects code and articulate feedback, the goal is to simulate this future reader for the pair. Critique their code base from the perspective of "I do not understand." Note Do not tell them how to fix it.

    For basically each code-inspection issue, they are expected to supply URLs to corresponding line ranges in github files. Hopefully this will also simplify the graders’ task and make it possible to get the grading done on time, even for large assignments.

    The pair may fail in various ways:

    if they

       

    we give

    fail to turn in a self-evaluation

       

    grade each code-inspection on a 60% scale.

    leave code-inspection blank,

       

    grade this issue on a 70% scale. That is, they can get at most 70% of the points for this issue.

    explain that they didn’t address an issue,

       

    given them 50% for this issue.

    misdirect for an issue,

       

    deduct 50% for accuracy.

    misdirect for two issues,

       

    they lose all points for accuracy.

  5. For every pair, there will be a form file in a the pair’s directory. The form file has this format:

    Here is the format for a code-inspection report:

        Pair: <pair-name>

        Commit: <commit-hash>

        Score: <points-earned>/<total>

        Grader: <name>

        

        free-text

    Graders write notes in the free-text area, fill in the first four fields, and push them back to master as they make progress.

  6. When the graders of a pod are done, the pod leader will spot-check the grading and inform Ben when the task is completed.

  7. Ben will push the feedback up to the students.

Graders should bring up any difficulties with their pod leaders first (interpreting the rubric, grading a specific case, meeting with a pair, or being late with grading).

Once the graders have evaluated the students’ work, the pod leaders will go check a selection from each grader to enforce a reasonable degree of consistency.
