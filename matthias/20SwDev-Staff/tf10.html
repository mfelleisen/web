Test Fest 10: How to Interpret the Scenarios

Client Tests: 20 Points

Two scenarios tested the students’ player-clients against my server. While the server is launched first—as a thread—the clients are launched as OS processes of course.

All clients are expected to behave properly. Every call to a client gets at least 10s, which is more than three times the promised time.

Test 1 runs five of their clients. A tournament of five normal players runs two games followed by a "final". One player wins.

Test 2 runs ten of their clients. A tournament of ten normal players runs three games followed by a "final". Three players tie for first place.

The most common mistakes you’ll see in the log:

  • a failure to send a proper JSON name

  • timing out on a call

  • not sending the specified JSON

Additionally you may see problems in the start-up script, e.g., it accepts the port on the command line and throws it away, using a default instead. It may also send bad JSON, which you will see in the log as an exception raised in my remote player and caught in the referee.

Server Tests: 80 Points

Eight scenarios tested the students’ server against my clients:

  • Two of them use five and ten ordinary players each, expecting the same results from their server that mine produced. See above. My players send prroper names, don’t time out, and otherwise do "good things."

    Even if their server does not satisfy the "accept a JSON string" protocol and instead accepts anything, these cases should still have worked for normal read operations.

    I do not know how their servers can fail in general in these scenarios. Each of them may suffer from a different error.

  • One test runs five normal players and one that fails to send a name. Their server should not accept this last player and simply ignore it.

  • Two tests don’t send well-formed JSON or valid JSON. Otherwise they use five ordinary players.

    If a server fails in this case, it will likely fail in one of two ways: (1) it doesn’t catch the exception and the system goes down or (2) it accidentally accepts the bad JSON and delivers a bad result. There are other, individual mistakes.

  • Three test cases launch some number of normal players and five misbehaving players that crash locally or go into infinite loops for the five simple client calls (start-tournament, playing-as, playing-with, setup, take-turn). One test includes a client that falls over on end-tournament and has to be turned into a non-winner.

    For this last case, my test harness is programmed to accept one of two results, which is why you will see things such as "expected result  [[0,1],[0,2]]."

Assessment

Timing out should not pose a problem for anybody who followed the "build the tree in a lazy fashion advice."

The server tests are also straightforward. Disobeying the connection protocol in small ways (not reading the string properly) works out. Half the tests are about the new kind of mistakes that Remote can inject and that can bring down a server without grace.
