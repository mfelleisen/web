Purpose Keep three points in mind. First, the grades are ok+, ok, ok-, and zero. Second, the students will vote on the meaning of these grades at the end of the semester. Third, the value of these grades is 30%—meaning their vote is devalued a lot.

The goal is to provide an assessment–grade and weak points—plus actionable feedback on how to improve as presenter or panelist.

Evaluating a Presenter The evaluation of a presenter involves two different aspects: the presentation/explanation of the code and the conversations with the panelists.

For the presentation/explanation aspect, consider the following questions after observing the code walk:

  • Did the presenter proceed in a “top down” manner? Clarify the overall purpose of each unit of code? Explain the structure of the code?

  • Did the presenter aim to present difficult pieces of code once the purpose & overview part was done? Or did the presenter try to show just the working pieces of code?

  • Did the presenter explain methods/functions on a line-by-line basis (bad) or on a holistic basis, diving into details as needed?

For the conversation aspect, ponder these:

  • Did the presenter answer basic/clarification questions easily? Are there any doubts about pair programming?

  • How did the presenter react to challenges? Did the presenter accept wrong criticism? Did the presenter reject valid criticism? When in doubt, did the presenter engage in a clarifying conversation?

Evaluating a Panelist For the panel, the focus is on the head reader and the assistant reader. If the secretary asks astute questions, take note.

For both, ask yourself the following:

  • Did the panelist ask the basic, design-recipe questions? Data interpretations? Purpose? Signature? One task per function or composite functions? Unit tests? Magic constants?

  • Did the panelist find a bug? Is the bug minor/easily fixed? Does it require serious work?

  • Did the panelist expose a conflict with the specification? A design flaw?

  • How well did the panelist articulate questions?

The head has an additional task, so there are two more questions to consider:

  • Did the head direct the presentation when the presenter lost focus? Tried to avoid the complex parts?

  • Did the head prune conversations when needed? This may not always be needed.

Preparing the Postmortem Session If you have time between class and the postmortem meeting, try to connect your notes with the above questions about the presenters and panelists:

  • Highlight the two most astute questions a panelist asked.

  • Assess the engagement of each panelist.

  • Highlight the presenter’s most appropriate responses.

  • Assess the attitude of each presenter.

If there are any doubts about the pair programming aspect of the presenters, try to write down a pro/cons list of what speaks for or against pair programming.

Special Notes Megan and Ryan should take turns focusing on mostly panelists xor mostly presenters. The first one should sit in the front to observer the panelists closely; the second one should sit to get a good sense of a presenter’s command of the code base.
